# Change: 多角色任务状态管理

## Why

当前任务系统对负责人(Assignee)、校核人(Reviewer)和审查人(Reviewer2)三个关键角色使用统一的任务状态(Task.status)进行管理，无法独立追踪每个角色的处理进度。当任务在某一环节被驳回时，系统无法清晰区分是哪个环节出了问题，也无法准确知道每个角色的工作进展。

此外，工作量单位为"天"，不够精确，且所有用户都能看到工作量信息，存在权限控制不当的问题。

## What Changes

### 1. 任务状态字段拆分
- **新增** `assigneeStatus` 字段 - 负责人工作状态
- **新增** `checkerStatus` 字段 - 校核人工作状态
- **新增** `chiefDesignerStatus` 字段 - 主任设计工作状态
- **新增** `approverStatus` 字段 - 审查人工作状态
- 每个角色状态的值：未开始、进行中、修改中、驳回中、已完成

### 2. 任务主状态调整

- **调整** `Task.status` 枚举值从 `['未开始', '进行中', '已完成']` 改为：
  - 未开始、编制中、修改中、校核中、审查中、已完成

### 3. 状态联动规则

- **新增** 自动状态计算逻辑：当所有角色的任务状态均为"已完成"时，任务主状态自动转换为"已完成"
- **新增** 驳回状态传播：任一角色状态变为"驳回中"时，主状态相应回退

### 4. 个人工作台增强
- **新增** 显示用户所属环节（负责人/校核人/审查人）
- **新增** 显示各环节的独立状态
- **调整** 任务列表增加状态列

### 5. 工作量单位与权限调整
- **调整** 工作量单位从天优化为小时（Workload 字段含义变更）
- **新增** 权限控制：仅班组长(LEADER)和管理员(ADMIN)可查看工作量信息
- **调整** 个人工作台工作量显示隐藏逻辑

## Impact

- **Affected specs**: task-pool（任务库）
- **Affected code**:
  - `types.ts` - Task 接口修改、RoleStatus 枚举定义
  - `dataService.ts` - 状态计算逻辑、数据迁移、权限控制
  - `TaskView.tsx` - 任务表单增加角色状态字段、状态显示调整
  - `PersonalWorkspaceView.tsx` - 个人工作台显示环节和状态
  - `Dashboard.tsx` - 统计图表可能需要调整
  - 相关数据需要进行迁移或初始化
